# Follow-Up Workflow Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the follow-up person's workflow from issued supplier PO through acknowledgement, reminder tracking, shipping documents, logistics, packing list, delivery order, invoice, payment reminder, and closure.

**Architecture:** Use `supplier_po_lines` as the item-level traceability anchor because each line already links supplier PO, buyer PO, quotation, quotation item, product, and manufacturer. Create follow-up records automatically from supplier PO lines, then progress each item through controlled stages with comments, reminders, required document gates, and generated final documents.

**Tech Stack:** Laravel API, Eloquent models/migrations, Vue 3 + TypeScript SPA routes/components, existing JWT auth/role checks, existing DOCX/PDF document service pattern.

---

## Current System Anchor

- `supplier_po_lines` is the source of truth for follow-up item creation.
- Each follow-up item must expose:
  - quotation reference
  - buyer PO number
  - supplier PO reference
  - buyer company/contact
  - supplier company/contact
  - product title and description
  - salesperson
  - assigned follow-up user
  - current stage
  - next reminder date
  - latest comment

## Attachment Findings

- Packing List sample fields: `PL-COR-394-OXY-25`, date, supplier, buyer, contacts, LPO number/date, item description, quantity, size, gross/net KG.
- Delivery Order sample fields: `DO-COR-394-OXY-25`, date, supplier, buyer, contacts, LPO number/date, item description, quantity, delivery place, terms, customer signature area.
- Invoice sample fields: `INV-COR-030-26`, invoice date, supplier/buyer, contacts, payment terms, currency, supplier/buyer TIN, supplier DO ref, supplier PO number, buyer LPO number/date, item pricing, VAT, bank details, customer signature.

## Phase 1: Follow-Up Core

**Files:**
- Create migrations for `follow_up_items`, `follow_up_comments`, and `follow_up_attachments`.
- Create models: `FollowUpItem`, `FollowUpComment`, `FollowUpAttachment`.
- Modify `SupplierPoController` so supplier PO creation creates one follow-up item per `supplier_po_line`.
- Modify supplier PO update so removed lines close/cancel their follow-up item only if no downstream work has started.
- Add API routes under `/api/follow-up`.
- Replace `/follow-up` placeholder with a real `FollowUpDashboardView.vue`.

**Behavior:**
- New supplier PO line creates follow-up item with status `awaiting_acknowledgement`.
- Follow-up dashboard shows only follow-up users' assigned items, while admin sees all.
- Admin can assign/reassign follow-up person.
- Salesperson can see linked follow-up status but cannot manage follow-up unless permission is added later.

**Tests:**
- Supplier PO creation creates follow-up rows for every line.
- Consolidated supplier PO creates separate follow-up rows for separate buyer quotations.
- Follow-up user can list assigned items.
- Unauthorized roles cannot access follow-up management endpoints.

## Phase 2: Reminder and Comment System

**Files:**
- Extend `follow_up_items` with reminder interval fields and `next_follow_up_at`.
- Add controller actions for setting reminder interval and adding comments.
- Add reminder UI to follow-up item detail.

**Behavior:**
- Follow-up person can choose interval: 1 week, 2 weeks, 3 weeks, 1 month, or custom date.
- Adding a comment records the communication and recalculates `next_follow_up_at`.
- Dashboard separates `due today`, `overdue`, and `upcoming`.

**Tests:**
- Comment resets reminder according to selected interval.
- Overdue items appear in overdue API response.
- Next reminder date is persisted and visible in item detail.

## Phase 3: Supplier Acknowledgement

**Files:**
- Add acknowledgement fields to `follow_up_items`.
- Add upload endpoint for acknowledgement file.
- Add acknowledgement panel in follow-up item detail.

**Behavior:**
- Status moves from `awaiting_acknowledgement` to `acknowledged`.
- Store acknowledgement received date, file, notes, and user.
- Keep acknowledgement as the first mandatory follow-up gate.

**Tests:**
- Cannot move to shipping before acknowledgement.
- Acknowledgement upload stores file metadata.
- Activity log records acknowledgement receipt.

## Phase 4: Delivery Responsibility and Shipping Document Gate

**Files:**
- Create `shipping_documents` table.
- Create `ShippingDocument` model.
- Add shipping workflow panel in follow-up detail.

**Required documents:**
- Supplier invoice
- Bill of lading
- Airway bill
- Certificate of origin
- Packing list data generated by ISC

**Behavior:**
- Follow-up user chooses delivery responsibility for the item: `isc`, `buyer_agent`, or `supplier`.
- The same document checklist appears after acknowledgement.
- Supplier invoice, bill of lading, airway bill, and certificate of origin are uploaded.
- Packing list is entered/generated by ISC with size and gross/net KG.
- The next logistics stage stays locked until all mandatory documents are complete.

**Tests:**
- Shipping cannot move to ETA until required documents are complete.
- Uploaded documents are linked to the follow-up item and supplier PO line.
- Delivery responsibility changes the next stage label but not the document gate.

## Phase 5: Packing List Generation

**Files:**
- Create `packing_lists` and `packing_list_items` tables.
- Create `PackingList` and `PackingListItem` models.
- Create `PackingListDocumentService`.
- Add API endpoints to create/download packing list DOCX/PDF.
- Add packing list UI inside the shipping panel.

**Behavior:**
- Packing list pulls supplier, buyer, contacts, buyer LPO, item description, and quantity from linked records.
- User enters size and gross/net KG.
- Generate references like `PL-COR-394-OXY-25`.
- Packing list completion marks the packing-list document requirement complete.

**Tests:**
- Packing list creates rows from selected follow-up items.
- DOCX contains reference, LPO, item, quantity, size, and gross/net KG.
- Packing list cannot be generated for items from unrelated buyer PO unless intentionally grouped by buyer delivery.

## Phase 6: Logistics and Goods Received

**Files:**
- Create `logistics_cases` and `logistics_events` tables.
- Create models and endpoints for ETA, warehouse receipt, buyer-agent handoff, and goods received.
- Add logistics timeline UI.

**Behavior for ISC delivery:**
- Documents complete -> ETA -> arrived -> delivered to ISC warehouse -> ready for DO.

**Behavior for buyer-agent delivery:**
- Documents complete -> documents sent to buyer agent -> waiting for buyer goods confirmation -> ready for invoice.

**Behavior for supplier delivery:**
- Documents complete -> supplier ETA -> buyer or ISC confirms goods received -> ready for DO/invoice according to delivery responsibility.

**Tests:**
- ISC flow requires warehouse received before delivery order.
- Buyer-agent flow can skip ISC warehouse and move to buyer received confirmation.
- All logistics events are timestamped and visible in timeline.

## Phase 7: Delivery Order Generation

**Files:**
- Create `delivery_orders` and `delivery_order_items` tables.
- Create `DeliveryOrder` and `DeliveryOrderItem` models.
- Create `DeliveryOrderDocumentService`.
- Add delivery order screen/panel.

**Behavior:**
- Delivery order pulls supplier, buyer, contacts, buyer LPO, item description, quantity, and delivery place.
- Generate references like `DO-COR-394-OXY-25`.
- Store issued date, delivered date, signed customer copy, and status.
- Signed DO is required before normal invoice creation for ISC-handled delivery.

**Tests:**
- Delivery order generation uses correct buyer-facing description.
- Signed DO upload changes status to `signed`.
- Invoice creation is blocked if signed DO is required but missing.

## Phase 8: Invoice and Payment Reminder

**Files:**
- Create `invoices`, `invoice_items`, and `payments` tables.
- Create models and invoice controller.
- Create `InvoiceDocumentService`.
- Add invoice screen/panel.

**Behavior:**
- Invoice pulls accepted quotation price, buyer PO, delivery order reference, payment terms, VAT, TIN, currency, and bank details.
- Generate references like `INV-COR-030-26`.
- If VAT rate is greater than 0 and VAT amount is 0, require reason or manager approval.
- Invoice has due date from payment terms.
- Payment reminder appears before/during overdue payment.
- Status moves: `invoice_created` -> `invoice_sent` -> `payment_pending` -> `paid` -> `closed`.

**Tests:**
- Invoice totals match accepted quotation price.
- VAT mismatch requires reason or approval.
- Due date is calculated from invoice date and payment term days.
- Payment record updates paid/balance status.

## Phase 9: Dashboard and Notifications

**Files:**
- Add follow-up dashboard API summary endpoint.
- Add dashboard widgets and notification badges.
- Add scheduled command for reminder generation.

**Behavior:**
- Dashboard shows awaiting acknowledgement, due reminders, overdue reminders, shipping document pending, ETA upcoming, goods received pending DO, delivered pending invoice, and invoice payment overdue.
- Notifications should be generated from database due dates, not only frontend calculations.

**Tests:**
- Scheduled reminder command creates notifications for due/overdue follow-up items.
- Admin sees all summary counts; follow-up user sees assigned counts.
- Salesperson does not see follow-up dashboard unless role/permission allows it.

## Recommended Build Order

1. Build Phase 1 and Phase 2 first so the follow-up person can see new supplier PO items and use reminders/comments.
2. Build Phase 3 next because acknowledgement is the first real operational gate.
3. Build Phase 4 and Phase 5 together because packing list completion is part of the shipping document gate.
4. Build Phase 6 after the document gate is stable.
5. Build Phase 7 and Phase 8 as document-generation phases.
6. Build Phase 9 after real statuses exist.

## First Implementation Slice

Start with:
- `follow_up_items`
- `follow_up_comments`
- auto-create follow-up items from supplier PO lines
- follow-up dashboard list grouped by supplier PO
- follow-up item detail with quotation, buyer PO, supplier PO, product, acknowledgement status, reminder interval, and comments

This gives the follow-up user immediate useful work without waiting for the full shipping/invoice workflow.
